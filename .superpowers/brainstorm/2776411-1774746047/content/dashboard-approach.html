Dashboard Revamp: Where should API health tracking live?

The current dashboard has 8 panels in a 3-column grid. We need to add API health/success tracking. How should we approach this?

A

Add API Health Panel to Existing Dashboard

Add a new "System Health" panel to the current grid — shows API success rates, recent failures, and status indicators alongside the existing trading panels. Minimal disruption to current layout.

Pros

  • Everything on one page
  • Quick to implement
  • Keeps existing layout intact

Cons

  • Dashboard getting crowded (already 8 panels)
  • Operational data mixed with trading data
B

Full Dashboard Redesign with Tabbed Views

Redesign the dashboard with tabs: "Trading" (current view), "System" (API health, costs, broker status), and "History" (trade log, sessions). Each tab is focused and uncluttered.

Pros

  • Clean separation of concerns
  • Room to grow each view independently
  • Better information hierarchy

Cons

  • Larger implementation effort
  • Loses "everything at a glance" benefit
  • More clicks to see all data
C

Compact Status Bar + Dedicated /health Page

Add a slim status bar at the top of the existing dashboard showing API health at-a-glance (green/yellow/red indicators per service). Link to a dedicated /system page with full details: call logs, success rates, error history, response times.

Pros

  • Best of both: glanceable health + deep detail
  • Dashboard stays clean
  • System page can be as detailed as needed

Cons

  • Two pages to maintain
  • Slightly more implementation work than Option A

My recommendation: Option C — the dashboard is already information-dense. A status bar gives you instant visibility, and the dedicated page gives you the detail when you need to debug.
